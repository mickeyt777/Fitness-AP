Production patterns this design bakes in

🗄️ Server is source of truth

Plan generation, stat aggregation and entitlement checks all run server-side. The client renders the response; one server fix reaches every client.

⚡ Optimistic UI + background sync

ViewModels update @Published state immediately, then reconcile on the async result. Start with workout logging — the highest-frequency write.

⚓ Anchor-based HealthKit reads

Persist HKQueryAnchor; pull only the delta each launch. Never read all-time data — it stalls startup and trips the rate limiter.

⚠️ The three architectural traps — pinned on the at-risk blocks above

1 God-object ViewModel

TodayViewModel will accrete plan + check-in + macros + steps + rest-day + trial banner. A step-fetch error then crashes the whole tab. Keep concerns split.

2 N+1 in the engine

progressWorkout() looks up each exercise in a loop. Free with an in-memory array — 18 DB reads where 1 would do once movements are a table. Batch the lookup.

3 Cardio double-count

A walk logged by HealthKit AND by chat doubles the day's totals and corrupts the step goal. Write the dedup rule as prose at the top of cardioService before coding it.
